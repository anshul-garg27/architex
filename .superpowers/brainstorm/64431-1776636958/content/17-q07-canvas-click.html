Deep-dive question 7 of 20 — canvas-to-lesson reverse sync. Small decision, big UX delight factor.

Q7 · In Learn mode, a user clicks a class on the canvas. What happens?

Currently designed: lesson scroll → highlights class on canvas (one-way). But what about the reverse? Canvas is read-only in Learn — clicks currently do nothing. That's either a delight opportunity or unnecessary scope.

🧠 USER BEHAVIOR TO IMAGINE
Mid-lesson on Factory Method. User is reading §5 (Code). Code references ConcreteCreatorA. User glances at canvas, sees that class, thinks: "Wait, when was this introduced?" They want to jump back to the lesson moment that first mentioned it.

Without reverse sync: they scroll up manually, search. Friction.
With reverse sync: click class → lesson scrolls to §3 (UML Reveal) where it was introduced. Magic.
A

Full bidirectional · click class → scroll lesson automatically

Clicking a class immediately scrolls lesson to the first/most-relevant section that mentions it. Canvas highlighting + scroll animate together.

Pros: Feels magical. Explorative learners love it.
Cons: Authoring: every class needs "introducedInSection" metadata. Ambiguous when class is mentioned in 3+ sections — which wins? Click during active scroll-sync = animation conflict.
B

One-way only · canvas click does nothing

V1 keeps it simple. Lesson drives canvas. Canvas is purely display in Learn mode. Cursor stays as default (not pointer).

Pros: Simplest. Zero new data needed. Ships fast.
Cons: Missed delight moment. Canvas feels less interactive than it looks.
C
⭐ MY RECOMMENDATION

On-demand popover · click shows choices, doesn't force jump

Click a class → small popover appears with: class summary (1 line) + list of lesson sections that reference it + "Jump to section" links. User chooses to jump or not. Keeps interaction optional and explicit.

D

Hover = tooltip with class description. Click does nothing. Cursor stays default.

Pros: Safe middle ground. Information without action.
Cons: Hover doesn't work on mobile/touch. No way to jump back to lesson section.
💡 WHY C · POPOVER IS THE RIGHT ABSTRACTION
The ambiguity problem with Option A is real: if Creator is mentioned in §3, §5, and §6, which one "wins" on click? Picking the first one arbitrarily feels wrong. Picking the nearest to current scroll is magic-but-unpredictable.

C sidesteps this entirely — let the user decide. Show them the list, they click the section they want. This is also strictly better on touch (no hover).

Data cost is modest: each lesson section already has highlightedClassIds (the forward-sync data). Inverting this gives us classesMentionedInSection. Both indices fall out of the same source. No new authoring work.

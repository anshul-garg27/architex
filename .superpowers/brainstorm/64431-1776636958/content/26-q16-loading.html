Q16 · How is lesson content loaded into Learn mode?

Current LLD codebase has 686 KB of static content. Adding 8-section lessons × 36 patterns × Opus-quality writing → probably ~2 MB content total. DB-first means API calls per pattern. What's the loading strategy?

A

All upfront · ship the monolith

Current pattern — all 36 patterns in initial bundle. ~2 MB gz. Pattern switching = instant. First load = slow.

Pros: Dead simple. No API calls. Offline works for anonymous users.
Cons: First-load hurts. Bundle size = worse Core Web Vitals = worse SEO. Every user pays for content they never read.
B

Lazy per pattern · pure on-demand

Initial bundle = shell only (no lesson content). Each pattern fetched via GET /api/content/lld/pattern/:slug when clicked.

Pros: Tiny initial bundle. Great Core Web Vitals.
Cons: Every pattern click = 200-500ms API call. Feels slow on bad networks.
C
⭐ MY RECOMMENDATION

Tiered prefetch · instant + warm + cold

Three tiers based on likely access:

① INSTANT · 5 foundation patterns in initial bundle
Singleton · Factory Method · Observer · Strategy · Builder — the 5 first-wave patterns most users open first. Shipped with initial JS bundle. ~225 KB gz. Zero API call for the first 5 experiences — critical for first-impression perf.
② WARM · next 10 prefetched on idle
After initial page load, requestIdleCallback kicks off background fetches for next 10 patterns (wave 1 remaining + wave 2 start). User clicks → already in TanStack Query cache = 0ms. Invisible prefetch.
③ COLD · remaining 21 on-demand
Rare patterns (AI-Agent, Concurrency, advanced modern) fetched only when clicked. 200-400ms fetch on click. Loading skeleton during. TanStack Query caches for 30min.
④ OFFLINE · service worker for everything accessed
Service worker caches every fetched pattern. Re-opening an already-seen pattern = offline-capable. PWA-ready for commute reading.
ADAPTIVE PREFETCH
// On idle, prefetch based on user context:
if (userProgress.completedPatternCount >= 3) {
  // Prefetch next pattern in path
  prefetchPattern(getNextInPath(userProgress));
}
if (hovering(patternTile)) {
  // Hover-prefetch — pattern loads before click
  prefetchPattern(hovered.slug);
}
Pros: First 5 patterns instant (covers 80% of users' first week). Adaptive prefetch = "next" patterns warm when users reach them. Hover-prefetch catches the remaining. Offline + PWA as free bonus from service worker. Bundle size cut by 75%+.
Cons: More orchestration. TanStack Query config + service worker + prefetch hooks. ~8-10h extra.
D

Static export · pre-render every pattern as HTML

Next.js SSG generates 36 HTML pages at build. Each /modules/lld/learn/[pattern] is a static HTML with content embedded.

Pros: SEO gold. Every pattern is its own URL.
Cons: Every content edit → rebuild. Breaks DB-first philosophy from §4.5.
💡 WHY C · TIERED PREFETCH
First impressions matter most. First 5 patterns instant = fast Core Web Vitals, immediate engagement. If users bounce in the first minute, nothing else matters.

Next 10 on idle = invisible. While user reads Singleton, background fetches Factory/Observer/Strategy/etc. By the time they finish Wave 1, Wave 2 is warm.

Rare patterns = on-demand. Only 5% of users will ever open "Interpreter" or "Flyweight". Pay the API cost for those clicks, not on everyone's initial bundle.

Service worker gives PWA for free. You were going to build PWA eventually. Caching lesson content is the easy win. Offline reading on the subway = retention bonus.

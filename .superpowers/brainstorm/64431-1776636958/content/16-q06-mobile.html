Deep-dive question 6 of 20 — mobile & responsive layout. Architex is desktop-first today. What's the strategy when a user opens LLD on a phone?

Q6 · How ambitious is mobile support?

We have 4 modes. Each one has a different feasibility on small screens. Not every mode should pretend to be mobile-friendly — some experiences are fundamentally desktop activities.

Feasibility per mode · the honest truth

🔁
REVIEW · MOBILE-GOLD ⭐⭐⭐⭐⭐
Already minimal. One question + 3 options + rating buttons. Better on phone than desktop. This is your daily-driver mobile experience.
📖
LEARN · FEASIBLE WITH ADAPTATION ⭐⭐⭐
3-col breaks, but lesson content works on phone. Canvas becomes tap-to-expand; scroll-sync replaced with inline diagram cards. Reading + checkpoints work great.
⏱️
DRILL · FEASIBLE BUT COMPROMISED ⭐⭐
UML editing on touch is painful. Mobile drill = read-only history + restart on desktop nudge. Or build a "sketch on paper, snap photo" text-describe alternative (expensive, v2).
🎨
BUILD · DESKTOP-ONLY ⭐
4-panel layout + React Flow + drag-drop + 75+ nodes + properties editor. Pretending this works on mobile is worse than not shipping it. Show "Best on desktop" nudge with email-me-a-link.

🌟 HERO ON MOBILE

Four strategies

A

Full responsive everywhere · all 4 modes work on mobile

Invest heavily in touch UML editing, mobile drag-drop, virtual keyboard handling. Every mode works at every breakpoint.

Pros: No user left behind.
Cons: 2x the engineering. Build mode on phone will be a bad experience no matter what. Touch UML is genuinely hard (Miro's mobile UX is mediocre, Figma's mobile UX is read-only). You'd be fighting physics.
B

Mobile-first for Review only

Review becomes phone-native (it's 80% there already). Other 3 modes are desktop/tablet only. Mobile visitors to Build/Drill/Learn see "switch to desktop" screen.

Pros: Clean scope. Review = habit-forming mobile app.
Cons: Learn on phone is genuinely useful for reading on commute. Blocking it feels overly strict.
C
⭐ MY RECOMMENDATION

Tiered support · be honest about what works where

Each mode gets the support level that matches its nature. Don't fake it.

🔁 Review · mobile-first — optimized for phone. Big buttons, swipeable rating, vertical layout. This is the primary mobile experience.
📖 Learn · responsive — full 3-col on desktop, stacked vertical on mobile. Canvas becomes an inline diagram card embedded between lesson sections (not a sticky panel). Scroll-sync replaced with "tap to zoom". Checkpoints always inline.
⏱️ Drill · read-only on mobile — mobile shows drill history + problem catalog + "Solve on desktop" CTA with email-link. Active drilling is desktop/tablet only. (Tablet drill works OK — stylus on iPad is viable.)
🎨 Build · desktop-only with honest fallback — on mobile, show a preview of user's saved diagrams + message: "Build mode needs desktop for precise UML editing. Open on laptop?" + email-yourself-a-link button.
Pros: Each mode gets the right treatment. Review becomes a habit-forming mobile daily driver. Learn works on commute. No apology for Build being desktop — pros respect tool honesty.
Cons: Multiple layouts per mode. ~10-15h extra over desktop-only.
D

Desktop-only now · native mobile app later

Ship all 4 modes desktop-only. Defer mobile to a React Native app built specifically for Review + Learn on phone. Separate concern, separate roadmap.

Pros: Laser focus. Best desktop experience possible. Mobile done right later, not halfway now.
Cons: Months before mobile. Loses 40% of web traffic. FSRS habit loop needs mobile to work.
💡 WHY C · HONEST TIERED SUPPORT
Not every screen needs to do everything. Figma doesn't edit on phone — they display. Notion doesn't fully edit on phone — they view. VS Code doesn't run on phone at all. Tool-honesty is respected by pros; trying to fake full parity produces cursed UX.

Review-on-mobile is the killer daily-habit loop. 2-minute reviews on the train. Duolingo owns this territory. FSRS + mobile Review is your retention engine.

Learn-on-mobile is reading. Commute reading is enormous for learning platforms. Stack vertical, make canvas inline and tap-to-zoom. Checkpoints work great with touch. ~8h of extra work.

Build-on-mobile is a lie. UML editing with 75+ node types + edge routing + properties panel on a phone is cursed. Saying "open on desktop" with an email-me-a-link button is kinder than shipping a bad experience.

Drill-on-mobile is half-useful. History review works. Solving doesn't. Be honest.
IMPLEMENTATION HOOK
Existing hook use-media-query.ts already exists. New useLLDResponsive() returns { breakpoint, canEditUML, showDesktopNudge } and each mode layout reads it to decide rendering. ~5h to build the hook + responsive shell. Mode-specific mobile layouts (Learn, Review) add ~8h each. Drill read-only + Build nudge screen ≈ 3h each. Total ≈ 24-27h added to Section 5's estimate.

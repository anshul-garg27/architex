Q12 · When a drill starts, what's already on the canvas?

Your problems.ts data has starterClasses defined for each of 33 problems — skeleton classes with method stubs. Question: do we pre-load these, start blank, or give the user the choice? This affects interview realism vs practice accelerator tradeoff.

📦 WHAT'S IN problems.ts PER PROBLEM
interface LLDProblem {
  requirements: string[]; // 5-8 functional requirements
  starterClasses: UMLClass[]; // 4-10 skeleton classes with method stubs
  starterRelationships: UMLRelationship[];
  hints: string[]; // 5-7 progressive hints
  keyPatterns: string[];
  classCount: number; // expected count
}
Interview-realism argument

Real interviewer hands you a prompt, not starter code. Starting blank trains the actual skill: go from requirements → classes from nothing. Starter classes = training wheels.

Pattern-practice argument

For learners drilling a specific pattern (say Observer in Chat app), they don't care about re-creating User/Message classes. Starter classes let them focus on the pattern-relevant structure.

A

Always blank · pure interview simulation

Timer starts, canvas is empty. Only the requirements list is shown. User designs from scratch every time. starterClasses data becomes the reference solution baseline for grading, not user input.

Pros: Maximally realistic. Strongest interview prep.
Cons: Intimidating for first drill ever. Users abandon if they freeze staring at blank canvas.
B

Always starter · accelerated practice

Starter classes pre-loaded. User fills in methods, adds missing classes, sets up relationships.

Pros: Lower friction. Pattern-focused practice.
Cons: Unrealistic — real interviews are blank. Creates false confidence.
C
⭐ MY RECOMMENDATION

User picks at drill start · 3 modes per problem

When user clicks a problem, modal asks:

Design Parking Lot · how do you want to start?
🎯
Interview mode · blank canvas DEFAULT
Requirements + timer. Design from scratch. Realistic.
🚂
Guided mode · starter classes
Pre-populated skeleton · fill in methods + relationships. Lower stakes.
⚡
Speed mode · starter + no timer
Pure practice. No pressure. Grade records as "practice", not in SRS stats.
Pros: User autonomy. First-time users pick Guided; crammers default to Interview. Speed mode is a welcome "untimed practice" option no competitor offers. All 3 use same grading engine.
Cons: One extra click before drill starts. Solved by remembering user's default choice.
D

Difficulty-tiered · Easy = starter, Medium = partial, Hard = blank

Algorithmic decision per problem. Easy gets full starter. Medium shows 2-3 seed classes. Hard is blank.

Pros: Progressive challenge.
Cons: User can't override. Advanced learner doing easy problem gets free classes they don't want. Forces design philosophy on user.
💡 WHY C · 3 MODES PER PROBLEM
Different users, different drills. Someone prepping for a Meta interview in 2 weeks wants Interview mode (hard mode, timed, realistic). Someone learning Observer wants Guided (skip infrastructure, focus on pattern). Someone curious wants Speed (no pressure, just explore). C lets all three exist without forcing you to pick a default philosophy.

Speed mode is competitive moat. LeetCode has no "untimed pattern practice". Brilliant doesn't do interview prep. A "just let me design without pressure" mode is gold for learners stuck between Learn and Drill. Your grading engine runs either way — zero extra engine work.

Remember user default. Third-time user never sees modal — lands in their preferred mode. First-timer sees it once to make an informed choice. Modal is onboarding, not friction.

SRS stats separation. Speed mode grades don't count toward FSRS (no pressure = different signal). Interview + Guided both do. This keeps retention metrics honest.

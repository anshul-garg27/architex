State & routing — the concrete shape

Four questions: what state is new, where does it live, what persists, and how does URL sync work.

1 · Extend ui-store.ts

Two new fields. Mode lives here because it's UI-level state (not content, not progress).

// src/stores/ui-store.ts — additions marked with ⊕

export type LLDMode = "learn" | "build" | "drill";

interface UIState {
  // ... existing fields (activeModule, theme, panels, ...)

  ⊕ lldMode: LLDMode | null;
    // null = first visit, not yet chosen
  ⊕ lldWelcomeBannerDismissed: boolean;

  ⊕ setLLDMode: (mode: LLDMode) => void;
  ⊕ dismissLLDWelcomeBanner: () => void;
}
✓ Why null over default "build"
Null = we can distinguish first-visit from returning-user-who-picked-build. Without null, migrated users would silently land in Build and miss the welcome banner.
✓ Why separate dismissed flag
User might dismiss banner but not pick a mode yet. Banner should stay dismissed even before mode is explicitly set.

2 · Extend interview-store.ts

Drill-specific state goes here (not a new store). Interview-store already tracks challenge timer + attempts — drill is a natural extension. One cohesive "interview prep" store instead of two.

// src/stores/interview-store.ts — additions

interface ActiveDrill {
  problemId: string;
  startedAt: number;  // epoch ms
  pausedAt: number | null;  // non-null = paused (user switched modes)
  elapsedBeforePause: number;  // accumulator across pause/resume
  durationLimitMs: number;  // per-problem (easy: 20m, med: 30m, hard: 45m)
  hintsUsed: { tier: "nudge" | "guided" | "full"; usedAt: number }[];
  submittedAt: number | null;  // null = in progress
}

interface InterviewState {
  // ... existing: currentChallengeId, timeElapsedSeconds, etc.

  ⊕ activeDrill: ActiveDrill | null;

  ⊕ startDrill: (problemId: string) => void;
  ⊕ pauseDrill: () => void;  // called on mode switch
  ⊕ resumeDrill: () => void;
  ⊕ submitDrill: () => void;  // final, triggers AutoGrader
  ⊕ abandonDrill: () => void;  // user clicks "give up"
  ⊕ useHint: (tier: "nudge" | "guided" | "full") => void;
}

4 · URL ↔ store sync

A single hook, mounted at the LLD module level, handles bidirectional sync.

// src/hooks/useLLDModeSync.ts (new)

export function useLLDModeSync() {
  const router = useRouter();
  const searchParams = useSearchParams();
  const { lldMode, setLLDMode } = useUIStore();

  // 1. URL → store (on mount, on navigation)
  useEffect(() => {
    const urlMode = searchParams.get("mode");
    if (isValidMode(urlMode) && urlMode !== lldMode) {
      setLLDMode(urlMode);
    }
  }, [searchParams]);

  // 2. store → URL (on mode change)
  useEffect(() => {
    if (lldMode && searchParams.get("mode") !== lldMode) {
      const params = new URLSearchParams(searchParams);
      params.set("mode", lldMode);
      router.replace(`?${params}`, { scroll: false });
    }
  }, [lldMode]);
}
Why router.replace, not push: Mode-switching shouldn't create back-button entries. User hits "Back" once → returns to where they came from (not the previous mode). scroll: false prevents jarring scroll-to-top on sync.

5 · Mode switch side effects

When mode changes, each mode may need to clean up. This lives as a middleware-like function, not scattered useEffects.

// Inside ui-store.setLLDMode action:
setLLDMode: (newMode) => {
  const prevMode = get().lldMode;
  if (prevMode === newMode) return;

  // Exit side effects for the OLD mode
  if (prevMode === "drill") {
    useInterviewStore.getState().pauseDrill();
    useNotificationStore.getState().show({
      type: "info",
      text: "Drill paused. Resume when you switch back.",
    });
  }
  if (prevMode === "learn") {
    useProgressStore.getState().saveScrollPosition();
  }

  // Enter side effects for the NEW mode
  if (newMode === "drill") {
    const drill = useInterviewStore.getState().activeDrill;
    if (drill?.pausedAt) useInterviewStore.getState().resumeDrill();
  }

  set({ lldMode: newMode });
}

6 · Migration for existing users

Before this change ships, users have no lldMode in their localStorage. How do they experience the upgrade?

① OPEN /modules/lld
lldMode === undefined in localStorage → zustand hydrates as null.
② MODE RESOLUTION
No URL param + null in store → "Learn" mode + welcome banner shown.
③ FIRST PICK
User dismisses banner or clicks a path → lldMode set → persisted → they never see banner again.
Existing build-mode users preserved: Their canvas state is untouched (lives in canvas-store, not ui-store). When they click "Let me build" in the banner, they land in Build mode with everything intact.

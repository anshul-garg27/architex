Q20 · How do we actually ship the 4-mode system?

This is a significant UX overhaul. Existing LLD users open the module → suddenly see a completely reorganized interface. A bad rollout = support tickets, churn, "where did my workspace go?". Choose the safety-net.

⚠️ WHAT COULD GO WRONG
• Existing users lose their workflow (Build mode exists but their muscle memory breaks)
• First-visit banner gets accidentally shown to returning users → looks infantilizing
• Content for 26 patterns ships half-written in early pushes
• Drill mode has a grading bug that hits first 500 users before detection
• Review mode FSRS scheduling miscalibration affects retention metrics invisibly
• Mobile layouts break on a browser we didn't test
A

Big bang · ship when ready, flip the switch

Build all modes. Test internally. One deploy, everyone sees it. No flags, no cohorts.

Pros: Simple. No feature-flag infra. Clear launch moment.
Cons: Bugs hit everyone. Rollback = full deploy. No A/B data. First-day support load terrifying.
B

Parallel routes · /modules/lld vs /modules/lld-v2

Keep old LLD at existing URL. New 4-mode system at /modules/lld-v2. Both live for months. User chooses via banner.

Pros: Zero risk to existing users.
Cons: 2x code to maintain. Forked state trees. Migration eventually happens anyway.
C
⭐ MY RECOMMENDATION

Feature-flagged cohort rollout · phased ramp

Single codebase. lldV2Enabled flag. Cohort-gated rollout. Rollback via flag flip, no deploy.

WAVE 1 · internal
Week 0 · ~5 users
You + 2-4 teammates. Flag manually enabled for your user IDs. Stress-test all 4 modes, all edge cases from Q11 cross-mode table.
WAVE 2 · alpha opt-in
Week 1 · ~50-100 users
Banner in existing LLD: "Try the new Learn/Build/Drill/Review experience? [Enable beta]". Users opt-in. Feedback form prominent. Toggle to return to old UI anytime.
WAVE 3 · anonymous 100%
Week 2-3 · new anonymous visitors
All anonymous (logged-out) visitors get the new UI by default. No confused existing users. Monitor drill grade distribution, Review completion rate, churn signals.
WAVE 4 · authenticated 10% → 50%
Week 4-5 · ramp gradually
Hash userId → deterministic cohort. Start 10%, monitor metrics vs control. If healthy, ramp to 25%, 50%. Rollback = flip flag config, takes 30s.
WAVE 5 · 100% + kill switch
Week 6+ · full rollout
All users on new UI. Old code path stays for 4 more weeks behind a "revert to old" settings toggle (hidden) for anyone who really needs escape hatch. Then purge.
🚨 ROLLBACK TRIGGERS (auto-alert)
• Drill submission error rate > 5%
• Learn pattern completion rate drops > 20% vs control cohort
• Mode switch bug causing data loss reports > 3 in 24h
• FSRS review rating distribution shifts suspiciously (e.g., 80% "Again" = review broken)
Hitting any of these → auto-flag revert + page on-call
FLAG IMPLEMENTATION
// userPreferences.preferences.lld.v2Enabled || cohortRoll() || envOverride()
function isLLDV2Enabled(user) {
  if (user?.preferences.lld.v2Optout) return false;
  if (user?.preferences.lld.v2OptIn) return true;
  if (!user) return ROLLOUT.anonymous;  // always true post-wave 3
  return hashCohort(user.id) < ROLLOUT.authenticatedPercent;
}
Pros: Safety at every step. Real user data informs ramp decisions. Rollback without deploy. 4 weeks of dual-path code is finite and bounded. Opt-in alpha generates testimonials + feedback. Existing users never get surprised.
Cons: 4-6 week rollout calendar. Dual UI paths exist briefly (shared shell minimizes this). Flag infrastructure needs to be reliable.
D

Stealth drop · ship, don't announce, observe

Ship quietly. No announcement. Monitor closely. Add announcement 1 week later if metrics look good.

Pros: Low-key launch.
Cons: Still big-bang underneath. Existing users get surprise UI change with no opt-in. Support load still spikes.
💡 WHY C · PHASED RAMP WITH ROLLBACK
Every wave earns the next one. Internal proves it works. Alpha opt-in proves users like it. Anonymous proves new-user funnel is good. 10% authenticated proves it scales. 50% proves it doesn't break. 100% is just closing the loop.

Anonymous-first is clever. Logged-out users have no muscle memory → no confusion. They become the proof-of-concept cohort before you touch authenticated users.

Kill switches are non-negotiable. Hitting auto-rollback triggers means a problem was caught at 10% instead of 100%. That's the entire point.

4-week calendar is realistic. Section 5 estimated 58h for implementation. Add 4 weeks for rollout on top = ~2 months total from code-complete to 100%.

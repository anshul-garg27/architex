Deep-dive question 8 of 20 — "tinker" permission in Learn mode. Pedagogy vs curation tradeoff.

Q8 · Can users edit the canvas mid-lesson?

Learn mode's canvas is read-only today. But Manu Kapur's Productive Failure research (cited in your README) says hands-on experimentation boosts retention 2-3x. What's the right balance between curated and experimental?

⚖️ THE REAL TENSION
Curated lesson argument

Scroll-sync depends on canvas matching lesson expectations. If user moves classes, highlights end up in weird places. Lesson authored for specific UML — breaking that breaks the lesson.

Experimental argument

"What if I added another ConcreteProduct?" is the exact productive failure moment. Blocking it teaches learned helplessness. Brilliant lets you poke; Duolingo lets you try. Tool should encourage curiosity.

A

Strict read-only · explicit mode switch required

Canvas is inert in Learn (except for Q7's popover). To experiment, user clicks "Switch to Build" — gets a fresh Build workspace.

Pros: Zero state confusion. Lesson remains curated. Simple.
Cons: High friction for mid-lesson curiosity. Starting from blank Build canvas loses the lesson's UML context.
B
⭐ MY RECOMMENDATION

Tinker mode · temporary unlock with reset & fork-to-Build

Small "✏️ Tinker" button floats above canvas in Learn mode. Click → canvas becomes editable. Scroll-sync pauses. A persistent toolbar appears with [Reset to lesson] [Save to Build] [Done tinkering].

C

Always editable · scroll-sync adapts to user's canvas

Canvas is always editable in Learn. Lesson highlights try to match user's current classes (by name) — if user renamed or removed them, highlight fails gracefully.

Pros: Maximum flexibility. No mode sub-states.
Cons: Lesson authoring becomes hostile — every section must handle "what if class doesn't exist". Breaking canvas breaks lesson. Curation erodes.
D

Section-end CTAs only · "Try it in Build →" at end of each section

No mid-lesson editing. Each lesson section ends with a "Try it" button that opens Build mode with the section's final UML as starting state.

Pros: Explicit boundaries. No sub-states.
Cons: Curiosity is in-the-moment — "can I try this right now?". Section-end CTAs are too late. Loses the teachable instant.
💡 WHY B · TINKER RESPECTS BOTH MODES
Default path stays curated. 95% of users will never click "Tinker" — they'll scroll lesson, take checkpoints, move on. Their experience is identical to A.

Curious learners get an escape valve. The 5% who think "wait, what if..." click Tinker → poke at the canvas → click Done → lesson resumes exactly where they left off. Zero lost progress. Experiment discarded by default — aligns with ephemeral exploration mindset.

Save-to-Build is the killer secondary action. If a learner's tinker produces something interesting, one click preserves it. This is also the ONLY bridge from Learn to Build that preserves state — a natural pedagogical arc.

Implementation cost is modest. Tinker mode = toggle a canvasEditable flag + pause scroll-sync hook + persist snapshot before entering. ~6-8h.

Q11 · How aggressively do we preserve state across mode switches?

Four philosophies for the policy

A

Aggressive · preserve everything exactly

Every UI sub-state persists. Mid-typed answers, open dropdowns, scroll positions, timer millis. Returning user sees identical view to before switch.

Pros: Feels magical.
Cons: Preserving transient UI (open dropdowns across a mode switch) is uncanny-valley. Storage bloat.
B
⭐ MY RECOMMENDATION

Smart boundaries · work persists, UI state discards

The policy the table above implements: work (canvas, progress, timer, scroll) persists to DB. Transient UI (dropdowns, half-typed inputs) discards. Ambiguous cases (unsaved tinker, mid-submit) get explicit confirmation.

Pros: User trust — never silently loses work, never uncanny-resumes random UI. Explicit confirmation for edge cases respects user agency. Storage-efficient.
Cons: 3 dialogs possible (tinker save, drill mid-submit, drill grade). Must be designed well to not be annoying.
C

Confirmation-first · always ask before switch

Any mode switch triggers: "Switching to Build. Keep your Drill in progress? [Keep] [Abandon]". Explicit, conservative, slow.

Pros: Never silent loss.
Cons: Every mode switch hits a dialog. Switcher becomes slow. Pros hate this.
D

Clean-slate · modes are independent · only canvas survives

Switching modes is a reset. Only the durable canvas state (Build scratch, named diagrams) crosses modes. Everything else resets.

Pros: Simplest mental model.
Cons: Lose mid-drill timer on Review switch = infuriating. Lose learn scroll on Drill switch = "why did it reset?"
💡 WHY B · THE TABLE IS THE SPEC
The key insight: not all state is created equal. Work (canvas, progress, timer) deserves aggressive preservation because losing it is devastating. Transient UI (dropdowns, typing) should discard because preserving it feels weird. Ambiguous cases deserve user agency — confirm, don't decide silently.

Three dialog moments is the budget. Tinker unsaved, drill mid-submit, drill grade dismissal. Not 10. Not 0. Three is enough to respect the user, few enough to not annoy.

Three approaches to serve all 4 personas

You picked D — serve everyone. Here are three ways to actually architect that. Visual wireframes below. My recommendation is at the bottom.

1

Persona Gate → 3 Separate Routes

First visit hits a welcome gate. User picks path. That choice routes them to a dedicated URL: /lld/learn, /lld/build, /lld/drill. Each is a fully distinct UI optimized for its persona. Mode switcher in top nav for later changes.

Pros

  • Each mode optimized independently — clean
  • Clear intent = clear UX per persona
  • URL-addressable — shareable learn/drill links

Cons

  • Onboarding gate = extra click friction
  • 3 distinct UIs to maintain = 3x work
  • Current "/modules/lld" has to split — migration risk
Effort: 🔴 Large (4-6 weeks) · Risk: Medium · Future flex: High
2

One Adaptive UI (progressive unlock)

No persona gate. First visit = minimal canvas with one big CTA: "Try Singleton →". UI unlocks progressively as user completes walkthroughs. Advanced tools (Mermaid editor, SOLID spotter, code generator) appear after milestones. Same URL throughout.

PROGRESSIVE UNLOCK — same URL, UI grows

Pros

  • Zero onboarding friction — start fast
  • Single UI = single codebase to maintain
  • Feels magical — "app grew up with me"

Cons

  • Crammer (persona C) hates unlocks — wants full power now
  • Progress-gating = frustrating if it gates wrong
  • Hard to serve D's "I know this, skip me ahead"
Effort: 🟡 Medium (3-4 weeks) · Risk: High (UX design risk) · Future flex: Medium

⭐ MY RECOMMENDATION

Hybrid — Mode Switcher inside shared shell

One URL (/modules/lld). On first visit, shell shows a compact "How do you want to start?" banner with 3 modes: Learn / Build / Drill. Pick one — content pane switches. Mode switcher stays visible at top. Returning users land on last-used mode. Same shell, 3 content modes inside.

Pros

  • Preserves current Build mode untouched
  • Ship incrementally — add Learn first, Drill later
  • One URL = SEO / sharing stays simple
  • Mode switcher = persona C can jump to Drill instantly
  • First-visit banner = picks their path without force-gate
  • Remembers last mode = returning user feels personal

Cons

  • Shell complexity grows — must handle 3 modes
  • Learn and Build fight over canvas real estate
  • "Smart defaults" tuning takes iteration
Effort: 🟢 Medium (3-4 weeks, shippable in 2 phases) · Risk: Low · Future flex: High
🎯 WHY I RECOMMEND APPROACH 3
Approach 1 is "cleanest" on paper but forces a 3-way fork in the codebase — 3 distinct shells, 3 onboardings, 3 state trees. For a team of 1-2, that's quarters of work. Approach 2 sounds elegant but breaks for persona C (they don't want training wheels) and D (they don't want unlocks at all).

Approach 3 uses your existing shell (canvas + sidebar + properties + bottom panel) as the stage. The current UI becomes Build mode. Learn mode replaces the content pane with scroll-synced lesson (the WalkthroughPlayer you already built). Drill mode replaces it with the problem queue. Three modes, one shell.

Critically: the WalkthroughPlayer component already renders all 4 checkpoint types. So "Learn mode" is ~70% built — we mostly need to reposition it from the bottom panel tab to a dedicated mode layout. That's the highest-ROI move.

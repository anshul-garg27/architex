Q14 · How deep is keyboard support in each mode?

cmdk palette exists. useKeyboardNodeOps hook exists. KeyboardShortcutsDialog exists. Question: do we build mode-specific shortcut vocabularies, or rely on the palette alone?

⌨️ WHAT EXISTS IN CODEBASE
• CommandPalette.tsx with cmdk · useCommandBus, useRecentCommands
• useKeyboardNodeOps — node add/delete/duplicate in Build canvas
• KeyboardShortcutsDialog — help overlay
• useFocusTrap for modals
A

Baseline only · standard web interactions

Tab, arrows, enter, escape. No custom shortcuts. Rely entirely on mouse + cmdk palette for power actions.

Pros: Cheap. No conflicts to debug.
Cons: Wastes the palette infrastructure. Pros hate. Review-mode rapid rating via keys is essential (Anki precedent).
B

Palette only · cmd+K for everything

Everything goes through palette. No mode-specific vocabularies. Search "submit drill", "next pattern", "answer B" — palette handles it.

Pros: Zero memorization. One surface to maintain.
Cons: cmd+K + type + enter = 3 keypresses for every action. Review mode with 50 reviews = 150 keypresses vs 50 with direct keys.
C
⭐ MY RECOMMENDATION

Global shortcuts + mode vocabulary + palette fallback

Three tiers: global (always work), mode-specific (contextual, respect input focus), palette (the discovery + fallback for unmemorized actions).

GLOBAL · WORK EVERYWHERE
⌘1 Learn mode    ⌘2 Build    ⌘3 Drill    ⌘4 Review
⌘K Command palette    ⌘/ Shortcuts help    Esc Close modal
📖 LEARN MODE
J/K Scroll sections    ←/→ Prev / next pattern
1..4 Answer checkpoint option A/B/C/D    Enter Submit
T Toggle Tinker mode    ? Ask the Architect
🎨 BUILD MODE
# via existing useKeyboardNodeOps + new:
N New class    E Edit selected    Del Delete
⌘Z / ⌘⇧Z Undo / redo    ⌘S Save as diagram
F Fit to view    M Toggle minimap
⏱️ DRILL MODE
Space Pause/resume timer    H Open hint panel
⌘Enter Submit (with confirm)    ⌘G Give up (with confirm)
R Show requirements    [ ] Prev / next problem in queue
🔁 REVIEW MODE · speed-optimized · Anki-inspired
1..4 Answer option A/B/C/D    Space Reveal answer
A / H / G / E Rate FSRS: Again / Hard / Good / Easy
Enter Next review item    Esc Exit session
Discovery: Pressing ? in any mode shows a mode-specific shortcuts overlay. Palette remains the "search any action" fallback for unmemorized shortcuts.
Pros: Review mode runs at Anki speed (critical for daily habit). Drill pros submit with ⌘Enter. Palette still handles the long tail. Mode-specific means no conflicts (J in Learn = scroll; J in Build could mean something else — not collision).
Cons: Memorization burden. ? overlay mitigates. Existing hooks do most of the work — new code ≈ 8h for Review shortcuts + Drill shortcuts.
D

Vim-style leader sequences · gf, gn, dw

2-key sequences. G = go, N = new, D = delete. gf = fit to view, gn = next pattern, dw = delete class. For terminal-pilled users.

Pros: Vim devs love it.
Cons: 95% of users aren't vim-pilled. Steep learning curve. Overkill.
💡 WHY C · LAYERED KEYBOARD
Review speed is critical. 1-4 to answer, A/H/G/E to rate FSRS, Enter to advance. 3 keypresses per review × 50 reviews = 150 keypresses. Via palette = 250 keypresses. Multiplier matters for a habit product.

Mode-scoped shortcuts = no conflicts. J in Learn is scroll. J in Build is nothing. Context-aware bindings avoid the "global shortcut hell" of Gmail-style vocabularies.

Palette is the fallback, not the primary. Press ⌘K to find "how do I...?". Learn the direct key over time. Progressive power-user unlock.

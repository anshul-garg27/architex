Q5 · Where does the FSRS review experience live?

FSRS-5 schedules reviews at optimal spacing. When a pattern is "due", user needs a frictionless way to review it. But we have 3 modes already. Does review become a 4th mode, get embedded, or live somewhere else?

⏰ WHY REVIEW IS A DISTINCT ACTIVITY
Review ≠ Learn (not teaching new material) · Review ≠ Build (not free exploration) · Review ≠ Drill (not timed problem). It's short retrieval practice on already-completed material. 2-3 minutes, 1-3 patterns, checkpoint-style. Designed as a daily habit, not an occasional deep dive.

The question: does it deserve primary screen real estate, or does it ride inside another mode?
A

4th primary mode · "Review"

Add a 4th pill to the switcher. Review lives alongside Learn/Build/Drill. Badge shows "N due" count in real-time.

B

Embedded in Learn · "Due for review" section in sidebar

Learn sidebar shows a "Review" section at top with N due patterns. Click → opens that pattern in Learn mode with only checkpoint section auto-scrolled into view.

Pros: Keeps 3-mode system clean. Learn already has checkpoints.
Cons: Review is hidden inside Learn — other-mode users never see it. Doesn't build daily-habit visibility.
C

Modal flow · accessible from anywhere

Persistent "Review (3)" button in top chrome across ALL 3 modes. Clicking opens an overlay modal with review flow. Close → return to whatever mode you were in.

Pros: Context-preserving. Always accessible.
Cons: Modals feel interruptive. Button competes with mode switcher for chrome space.
D

Drill-embedded · due reviews become mini-drills

Due reviews surface as a "Quick Review" category in Drill mode. Solving them counts toward retention. Conceptually neat — retention = application.

Pros: Elegant theoretical fit.
Cons: Drill ≈ 20+ minute activity. Review should be ≈ 2 min. Wrong tempo.
💡 WHY A · MAKE REVIEW A PRIMARY MODE
Your product has already bet on FSRS. Your README calls out "99.6% superiority over SM-2". Your progress table has 8 FSRS fields. Your use-due-reviews.ts hook exists. You've invested in the science — now give it the UI real estate to become an actual daily habit.

Duolingo's whole business model is Review-as-primary. Their app IS a review app dressed up with lesson chrome. Users don't "go to Duolingo to learn" — they go because of streaks and reviews. That's the engagement flywheel you want.

The minimal Review layout is a feature, not a bug. Review mode strips away sidebars, canvases, panels — one question at a time, big buttons. This is exactly what works on mobile (Q13 later will tackle mobile adaptation — Review mode is the easiest to make mobile-first).

Badge mechanics compound. Red "3" on the pill is a gentle pull back to the app. That's a retention feature worth a little UI complexity.
REVIEW MODE · SPEC SNAPSHOT
Entry: Click 🔁 Review in switcher. If 0 due → shows "All caught up · come back tomorrow".
Session: Sequential 1-3 patterns. One random checkpoint per pattern. User rates retention: Again / Hard / Good / Easy → updates FSRS.
Time budget: < 2 min for 3 patterns. Short by design.
Escape hatch: "Open in Learn →" on any checkpoint if user forgets. Keeps review forgiving.
Completion: Session summary + streak increment. CTA: "Try Drill mode?" or "Done for today".
Reuses: use-due-reviews.ts (existing), quizQuestions data (existing), progress FSRS fields (existing).
New components: ReviewModeLayout.tsx, ReviewSessionCard.tsx, ReviewEmptyState.tsx, ReviewStreakCelebrate.tsx (≈ +8h effort).

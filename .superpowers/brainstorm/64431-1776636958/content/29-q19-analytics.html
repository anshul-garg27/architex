Q19 · What do we track in the LLD module?

activityEvents table already exists (append-only). Plus PostHog scaffolding. Plus Sentry scaffolding. The question is: what's the event taxonomy and who consumes it?

👥 EVENTS HAVE TWO AUDIENCES
Product team (you)

"Where do users drop off? Which patterns are hardest? Does Review mode get used?" — funnel + retention analytics via PostHog/ClickHouse.

User themselves

"How much time have I spent on Observer? What's my drill pass rate this month?" — personal dashboard reads from same event stream.

A

Minimal · page views + logins only

Just Next.js analytics + Clerk login events. Nothing custom.

Pros: Easy. No new code.
Cons: Zero insight into in-app behavior. Can't improve what you can't measure.
B

Core events only · ~8 key moments

Mode switches, pattern completions, drill submissions, achievements. That's it.

Pros: Simple.
Cons: No funnel depth. Can't see where in Learn mode users drop off.
C
⭐ MY RECOMMENDATION

Full funnel · ~25 events with typed schema · dual-purpose

Typed event catalog. Each event writes to activityEvents (owned by user, drives personal stats) AND mirrors to PostHog (aggregate funnels).

CRITICAL FUNNELS THIS ENABLES
• Learn completion funnel: opened → scroll 25% → scroll 75% → all checkpoints → completed
• Learn → Drill conversion: completed pattern → drill started on same pattern
• Drill success funnel: started → hint used → submitted → graded ≥ 70
• Review retention: session started → rated item → session completed
• Learn → Mastered: opened → completed → drill pass → FSRS review → Mastered tier
• Abandonment signals: which step users drop at, which patterns are abandoned most
Pros: Every meaningful action captured. Dual-purpose (product + user dashboard). Reuses existing activityEvents — no new schema. TypeScript-typed event builders prevent drift. Each event < 1 KB so log size is fine.
Cons: 25 events = 25 call sites to wire. ~6-8h for full instrumentation.
D

Everything · including hover, focus, scroll positions

Heatmap-level tracking. Every interaction.

Pros: Rich data.
Cons: Massive storage cost. Privacy concerns. Noise drowns signal. Needs dedicated analytics infra at 100K+ DAU.
💡 WHY C · TYPED EVENT CATALOG
25 events is the sweet spot. Enough to answer every product question (funnels, drop-off, time-on-task, mode distribution, mastery velocity). Few enough to stay intentional and typed.

Dual-purpose saves infrastructure. Same event stream feeds your PostHog dashboards AND user's personal dashboard. activityEvents already designed for this.

TypeScript-typed event builders matter. Create src/lib/analytics/lld-events.ts with a function per event — trackPatternOpened(patternId, fromMode). Compile-time safety, refactor-safe, IDE autocomplete. Prevents the "what did we name that event again?" debt.

Q17 · Anonymous user signs up. What happens to their local progress?

Clerk is optional — a user could read 8 patterns, pass 3 drills, then decide to sign up. All of that is in localStorage. How do we handle the moment of account creation?

🎭 THE 4 SCENARIOS
S1: Anonymous builds state → signs up for first time. localStorage has data, DB has nothing.
S2: Anonymous → logs into existing account from another device. localStorage has local session; DB has history.
S3: Anonymous has no state → signs up. localStorage empty, DB empty. Clean start.
S4: User logged in → logs out → uses anonymously → logs back in. Mixed state.
A

Manual · user clicks "Sync my progress"

After login, a banner: "Bring your progress along? [Sync now] [No thanks]". User explicitly triggers migration.

Pros: Explicit.
Cons: User might dismiss and lose progress. Banners get ignored.
B

Silent auto-merge · always migrate

On login, automatically POST localStorage state to /api/user-preferences/lld/migrate. No user action. Toast: "Welcome! Your progress is saved."

Pros: Zero friction. Never lose work.
Cons: In S2 (existing account on new device), silent merge could OVERWRITE server state with stale local state. Data loss risk.
C
⭐ MY RECOMMENDATION

Smart hybrid · auto-merge when safe, confirm when conflicting

Server runs a 3-case decision tree on login:

CASE 1 · localStorage has data, DB empty (S1 · first sign-up)
Silent auto-merge. Welcome toast: "Welcome! Your 5 patterns of progress have been saved to your account." No decision asked — clearly the right move.
CASE 2 · localStorage AND DB both have data (S2 · conflict)
Confirmation dialog.
"We found 5 patterns you worked on in this browser, and 12 patterns saved to your account. What would you like to do?"
[Keep account · merge this browser's patterns if any are newer] [Discard browser progress]
CASE 3 · localStorage empty OR DB has everything (S3 · clean)
Nothing to do. Just log in. No toast needed.
CONFLICT RESOLUTION · "KEEP ACCOUNT · MERGE IF NEWER"
For each pattern:
  • If DB has progress but localStorage doesn't → DB wins (no change)
  • If localStorage has progress but DB doesn't → local wins (insert to DB)
  • If both have progress → higher completion tier wins (mastered > completed > introduced)
  • For equal tiers, latest `updatedAt` wins
Pros: Never loses work without consent. Clear UX for each scenario. Smart merge rules (tier-based, not timestamp-only) preserve progress. S2 risk eliminated.
Cons: 3 code paths to maintain. The merge algorithm needs careful testing for conflict case.
D

Always start fresh · localStorage discarded on login

On login, localStorage state is discarded. User starts from DB state only.

Pros: Simplest code.
Cons: Brutal. Users who built anonymous state lose it. Huge churn moment.
💡 WHY C · BE SMART ABOUT CASE 1 vs CASE 2
Case 1 is the important scenario. Most anonymous signups are first-time users. Silent merge feels magical: "my progress followed me". Never interrupt that moment with a dialog.

Case 2 is the dangerous scenario. Existing user logs in on a new device that happens to have stale localStorage. Blind merge = data loss. Confirmation dialog is the right safety rail — rare enough to not annoy, critical enough to show.

Tier-based merge wins over timestamp. If user on laptop completed Singleton (mastered tier), and mobile localStorage has only "introduced", timestamp could pick mobile if the phone's clock updated later. Tier-based picks the better state. Monotonic progress is preserved.

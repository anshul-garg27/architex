Q55 · Simulation clock & timestep model

Chaos drill says "5 minutes". In what time does that unfold?

OPTION A · WALL-CLOCK 1:1
"5 minutes" = 5 real minutes. Chaos events fire on wall-clock timer. Easy to reason about. But boring for long time-scale things (3-month cost forecast).
⭐ OPTION B · DUAL-CLOCK
Two clocks: real-time (for chaos drills, interview timer) and sim-time (for forecast, evolving campaigns — 3 months compressed into 3 min). User-aware labeling: "sim-time: 2 months elapsed · real-time: 1:47".
realtime: 0:01:47 │ sim-time: 2mo 7d
speed: 30 sim-days / real-min
OPTION C · FIXED TIMESTEP
Sim advances in discrete ticks (e.g. 100ms). All metrics roll at tick boundary. Deterministic; easy to replay. But coarse for sub-100ms chaos events.
OPTION D · VARIABLE TIMESTEP + DILATION
Fixed physics tick (100ms) but rendered frames adapt to user's rAF rate. Slow-mo (0.25x) = tick still 100ms but played over 400ms. Replay deterministic. Most engineering but highest fidelity.

Q56 · Metrics engine — how are p50/p95/p99 computed?

Latency percentiles need the right data structure. Naive arrays blow up memory at 1000 rps. What ships?

OPTION A · SLIDING ARRAY (today)
Last-N samples kept in a bounded array. Sort to compute percentiles. Simple but O(n log n) per sample. What we have today.
⭐ OPTION B · HDR HISTOGRAM
HDR Histogram (industry standard — used by Prometheus, Netflix, Datadog). O(1) recording, log-linear memory (~2KB fixed for full range). Accurate p99.99 up to hours of range. True production-grade.
npm: hdr-histogram-js (~2KB per histogram)
record(latencyMs) → O(1)
percentile(99.9) → O(bucketCount)
OPTION C · T-DIGEST
T-Digest (used by Cassandra, Kafka). Good accuracy at extreme percentiles. Mergeable across workers. Marginally more code than HDR but better for streaming systems.
OPTION D · RING-BUFFER + HDR
HDR histogram for live p-values + raw ring-buffer (30s of samples) for the time-scrubber replay. Dual structure. Covers both use cases at ~10KB per node.

Q57 · Load-model fidelity — how realistic is the traffic?

"200k rps" is a number. But what's the shape? Uniform? Bursty? Poisson?

① UNIFORM (naive)
Constant rate. Fails realism but simple. Only useful for initial learning modes.
② POISSON PROCESS ⭐
Inter-arrival times follow exponential distribution. Matches real-world random arrivals. Standard queuing-theory assumption. Cheap to compute.
nextArrival = -Math.log(1 - Math.random()) / λ
③ DIURNAL CURVES ⭐
Rate varies with time-of-day. Matches real traffic: 3 AM trough, noon peak, evening surge. Per-region curves (shifted by timezone). Essential for forecast mode.
④ BURST / FLASH CROWDS ⭐
Viral spike modeling — exponential ramp-up then decay. Used in "Reddit-hug-of-death" scenarios. Mathematically: Pareto-distributed burst events.
⑤ HOT-KEY DISTRIBUTION ⭐
Zipfian access pattern — 20% of keys get 80% of traffic. Teaches why consistent hashing breaks, why hot partitions form. Critical for sharding lessons.
⑥ USER-SEGMENT MIX
Mobile · Desktop · Bot · API-client — each has different request shape (size, frequency, error tolerance). Mix them realistically.
⑦ PER-ENDPOINT PROFILE
GET /feed has different rate + cache-hit ratio than POST /upload. Realistic when user designs >1 API.
⑧ REAL-WORLD TRACE REPLAY
Ingest anonymized real traces (Alibaba 2018, Google 2011, Meta F4) — replay against user's design. Ultra-realistic. Authoring cost per trace ~8hr.

Q58 · Cascade propagation & failure math

When cache misses → DB hot → retries amplify. This chain is the signature learning moment. How is it modeled?

OPTION A · SCRIPTED
Chaos events have pre-written cascade lists: "cache-evict → spike db-cpu". Simple. Lies to physics when user's design differs.
⭐ OPTION B · PHYSICS-DRIVEN + HYSTERESIS
Each node has saturation curves (latency goes exponential past 80% CPU), failure probability proportional to saturation, recovery hysteresis (can't recover instantly), circuit breaker state machines on edges. Cascades emerge from math — not scripts.
Node model:
cpu = load × tailFactor(load, capacity)
p99 = base + exp(cpu/capacity - 0.8) × 50ms
errorRate = max(0, (cpu/cap - 0.95) × 10)

Circuit breaker:
state: closed → open → half-open → closed
triggers: errorRate ≥ 0.5 over 10s window
OPTION C · QUEUING THEORY
Each node = M/M/c queue. Little's Law: L = λW. Exact latency for load ≤ capacity. Breaks down at saturation (real systems do too).
OPTION D · B + C HYBRID
Queuing math up to 80% load (accurate), saturation curves + circuit breakers past 80% (realistic). Best-of-both. Teaches correct math + correct failure behavior.

Q59 · Rendering engine — canvas performance & visual fidelity

1000+ nodes + animated data flow + node breathing + cascade arrows — at 60fps. How?

OPTION A · SVG + REACTFLOW (today)
Each node is a DOM element. Fine up to ~100 nodes. Breaks at 500+ with animation. What LLD uses.
⭐ OPTION B · HYBRID · SVG NODES + CANVAS PARTICLES
Nodes stay SVG (interaction, accessibility). Data-flow particles on a <canvas> layer underneath. Node-breathing via CSS transform (GPU-accelerated). Scales to 500-1000 nodes + 10k particles at 60fps.
OPTION C · PIXI.JS / WEBGL
Full GPU-accelerated canvas. 10k+ nodes possible. Loss: accessibility, text-selection, DOM events need custom layer.
OPTION D · B + LOD ZOOM
Hybrid (B) + Level-Of-Detail: zoomed-out shows simplified node (dot), zoomed-in shows full metrics. Matches semantic-zoom from Q49.

Q60 · Edge routing algorithm — beyond LLD's A*

LLD uses A* for class-diagram edges. SD has more diagram types → more routing needs.

① A* WITH CONTAINER PENALTY ⭐
Extension of LLD's A*. Cost function adds penalty for crossing region/zone containers. Teaches you: "your edge shouldn't cross AZ boundaries invisibly." For architecture diagrams.
cost(a,b) = dist + crossContainer×50 + crossNode×∞
② ORTHOGONAL ROUTING ⭐
Right-angle edges only (Manhattan routing). For deployment + network topology diagrams. Algorithms: orthogonal graph drawing + libavoid-style.
③ BEZIER FOR ASYNC ⭐
Smooth curves for async/streaming edges. Dashed, animated flow. Distinguishes from sync calls visually.
④ FORCE-DIRECTED (mesh diagrams) ⭐
d3-force for service-mesh + dependency graphs. Nodes repel, edges attract. Self-organizing. For the dependency-graph diagram type.
⑤ EDGE BUNDLING
Parallel edges with similar source/target → bundle into a thicker beam. Reduces visual noise in high-connectivity designs.
⑥ CROSSING MINIMIZATION
Incremental re-layout when user adds edges — minimizes crossings. OSage-style algorithm. Higher effort, notably cleaner visuals.

Q61 · Auto-layout per diagram type

LLD uses Dagre (layered). SD has 10 diagram types — one algorithm won't serve all.

① DAGRE (hierarchical) ⭐
Layered DAG. For architecture + data-flow. Same as LLD.
② ELK (layered + orthogonal) ⭐
Eclipse Layout Kernel. More options than Dagre. Handles deployment + network topology. ~50KB + webworker.
③ D3-FORCE (force-directed) ⭐
Service mesh + dependency graph. Organic. Live updates as user interacts.
④ SWIMLANE (columnar) ⭐
Custom — columns per team/owner. Sequence diagram too (time-ordered). Small, in-house.
⑤ RADIAL (dependency center)
Pick a node, others orbit by distance-from-it. For "blast-radius" views.
⑥ INCREMENTAL RE-LAYOUT ⭐
User adds a node → only affected sub-graph repositions. Surroundings stay stable. Anti-jitter. Hard engineering but critical UX.

Q62 · Deterministic replay + branching

From Batch 4 we locked Replay & What-If-Branching. The engineering to make those possible.

OPTION A · EVENT LOG ONLY
Store the event stream (chaos events, user actions). Replay runs engine forward. Deterministic if engine is pure. Memory-cheap.
⭐ OPTION B · SEED + EVENT LOG + KEYFRAME SNAPSHOTS
Seeded RNG + event log + full-state snapshots every 30s. Replay seeks to nearest snapshot, replays forward from there. Fast scrubbing. What-If branching = fork a snapshot + diverge.
sim-state: { rng-seed, event-log, snapshots[] }
replay(t) = snapshots[floor(t/30s)] + replayEvents(t%30s)
branch(t) = fork snapshot at t; new rng-seed
OPTION C · FULL STATE LOG
State snapshot every tick. Highest fidelity. Memory prohibitive for long sims (MB per minute).
OPTION D · B + SERVER STORAGE
Keyframes stored in DB (Neon Postgres JSONB) for shareable replays. Cross-device. Peer-review workflow unlocked.

Q63 · Request tracing visualization

"Watch one request travel through the system." Under the hood, it's OpenTelemetry spans in a tree.

OPTION A · ANIMATED PARTICLE
Dot moves through graph. Pretty. Low information density.
⭐ OPTION B · SPAN TREE · WATERFALL
Jaeger/Honeycomb-style waterfall timeline. Each bar = one service span. Shows parallelism vs serialization. Clickable → zoom to service.
gateway ████████████████ 16ms
└─auth ██ 2ms
└─app ████████████ 12ms
└─cache █ 1ms miss
└─db █████████ 9ms ⚠ slow
OPTION C · FLAME GRAPH
Vertical flame — aggregated spans across many requests. Shows hot paths. Advanced user feature.
OPTION D · A + B TOGETHER
Animated particle on canvas (A) + synchronized waterfall panel (B). Click span → particle jumps. Maximum signal.

Q64 · Multiplayer / collaborative simulation

From Batch 10 we locked "shareable artifacts". This is the live-simulation extension.

OPTION A · SPECTATOR ONLY
Share-link viewers watch a sim live (WebSocket relay). Can't interact. Simple infra, limited moat.
⭐ OPTION B · PAIR MODE · CO-OP
Two users, one canvas, shared engine. Either can interact. CRDT-based state sync (Yjs). Cursor presence. WebSocket. Pairs with "Live Pair Session" from Q41.
OPTION C · VS MODE · ATTACKER vs DEFENDER
One user designs. Other user picks chaos events from the 73-catalog and fires them. Timed rounds. "Can you keep your design alive?" Gamified. Highest engagement.
OPTION D · A + B + C
Full collab stack. Spectator · Co-op · VS. Progressive ship.

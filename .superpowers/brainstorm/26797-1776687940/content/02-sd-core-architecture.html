Q1 · How many modes? Same 4 as LLD, or add a 5th "Simulate"?

LLD locked on Learn / Build / Drill / Review. System Design's unique asset is the simulation engine — already 34 files of infrastructure. Does simulation deserve its own mode, or is it embedded within Build?

A

Same 4 modes · simulation lives inside Build

Build mode gets a "▶ Simulate" toggle. You design, you flip the switch, traffic flows. Consistent with LLD.

Pros: Cross-module consistency · simpler nav · fewer URL modes to manage. Cons: Buries the moat. Simulation is the differentiator; hiding it in Build under-sells it.
B
⭐ MY RECOMMENDATION

5 modes · Learn / Build / Simulate / Drill / Review

Simulate becomes its own primary mode. When you enter Simulate, you get a reference architecture (or your own design imported from Build) + chaos controls + live metrics + narrative engine. Simulate is where you learn by breaking.

📖 Learn
🎨 Build
⚡ Simulate
⏱ Drill
🔁 Review
Pros: Surfaces the moat. "See it break" is a distinct activity from "design it". Chaos storytelling needs its own real estate. Cons: Breaks cross-module consistency (LLD has 4). ⌘5 shortcut added.
C

4 modes but swap Drill for Simulate

For System Design specifically: drop "Drill" as a standalone mode, merge drill problems INTO Simulate ("design the architecture for Netflix, then simulate"). Simulate IS the drill for System Design.

Pros: 4-mode parity with LLD. Elegant — simulation-as-grading. Cons: Conflates two distinct activities. "Quick drill" (blank canvas, timer) is different from "run my design under load".

Q2 · Content structure · 36 problems (like LLD), or concepts library, or both?

LLD = 36 patterns. System Design's parallel unit is ambiguous. Three options:

A

Problem-centric · 36 canonical system design interviews

"Design Netflix · Design Uber · Design Twitter · Design WhatsApp · Design Dropbox · etc." 36 canonical problems in 8 waves. Each problem teaches the concepts needed by appearing IN context. Concepts are implicit, not standalone.

Pros: Symmetry with LLD's 36 patterns. Problem-first matches how interview prep works. Memorable names. Cons: Concepts repeat (load balancer appears in 30+ problems). No "one place" to learn load balancing deeply.
B

Concept-centric · ~40 core concepts, each deeply taught

Consistent hashing · CAP theorem · Rate limiting · CDN · Leader-follower replication · Pub-sub · Event sourcing · Idempotency · etc. Each concept gets a Learn-mode lesson. Problems are practice arenas that combine concepts.

Pros: ByteByteGo approach. Concepts get proper treatment. Searchable. Cons: Less emotionally memorable than "Design Netflix". Can feel academic.
C
⭐ MY RECOMMENDATION

Both · 40 concepts (atoms) + 30 problems (molecules)

Concepts are the teachable atoms (load balancer, consistent hashing). Problems are molecules that compose concepts under realistic constraints. Lesson view prioritizes concepts; Drill view prioritizes problems. A problem page links to every concept it uses. A concept page shows every problem that uses it.

Q3 · What do we do with the 53 existing templates?

The current template system has 53 pre-built architectures. They're unnamed scenarios without lessons attached. They need a fate.

A

Keep + catalog · rename as "reference architectures"

Each of the 53 templates gets metadata (name, 1-line description, which problem it solves). Accessible via Build mode's "Browse templates" button. No lesson, just a starting point.

B
⭐ MY RECOMMENDATION

Elevate · templates become problem solutions

Each of the 30 problems gets 2-3 "canonical" solutions adapted from existing templates. User in Drill mode designs their own. Learn mode shows one canonical. "Compare" mode shows your design vs canonical. The 53 templates get curated down to the best 60-90 solution variants (2-3 per problem).

Pros: Templates now serve pedagogy. User sees "5 ways to design Netflix" · canonical vs minimal vs enterprise. Cons: Content curation work (rename / document each).
C

Retire · start fresh

Don't carry forward. Design new reference solutions tied to the 30 problems from scratch.

Authoring protocol · parallel to LLD

Claude Opus 4.7 writes all 40 concepts + 30 problems · 1000% quality bar
Same brand voice, same 6-step teaching sequence as LLD, adapted for system design context:

Concept lessons (40 × 8-section): Hook → Analogy → Architecture reveal → Checkpoint → Code/config example → Tradeoffs → Summary → CTA

Problem lessons (30 × extended structure): Real context (why does this exist? what scale?) → Requirements capture → Naive solution → Constraints that break naive → Design evolution → 2-3 canonical solutions side-by-side → Tradeoff matrix → Interview Q&A

Total content volume: 40 concepts × ~1500 words = 60k words · 30 problems × ~3000 words = 90k words · plus code samples, diagrams, cross-links. Opus time: ~200-300 hours over 2-3 months (authoring cadence, with human review).

Each concept + problem gets these additional formats

📊 Scaling numbers
QPS targets, storage estimates, bandwidth math. "Netflix serves 100B requests/day. That's 1.2M QPS peak."
💥 Failure war stories
Real outages where this concept/problem failed. AWS S3 2017, Cloudflare July 2019, etc.
🎙 Engineering blog deep-links
Curated links to Netflix Tech Blog, Uber Engineering, Stripe Engineering articles on the topic.
🧪 Simulation scenario
One-click "Simulate this in the wind tunnel" — spins up reference architecture, pushes traffic, shows what breaks.
💰 Cost model
AWS cost estimate for reference solution at 1M / 10M / 100M users. Teaches cost-aware architecture.
🎭 Interviewer questions
Top 10 probing questions interviewers actually ask per problem. For Netflix: "how do you handle VP9 vs H.264 device compat?"
💡 KEY DIFFERENCE VS LLD
LLD content is theory-heavy — 36 patterns that apply anywhere. System Design content is context-heavy — every design involves scale numbers, real companies, actual tradeoffs. A concept isn't "load balancer the abstract idea" — it's "NGINX at 100k QPS breaks when X, here's why, here's how Netflix solved it".

This is why System Design content takes 2-3x longer to author than LLD. Every claim needs numbers. Every design needs a failure mode. Every tradeoff needs "we know this because Company X hit it in 2019 and wrote a post-mortem".

Opus can do this — the research exists on Netflix Tech Blog, Uber Engineering, ACM Queue, etc. But the authoring is research + synthesis, not just creative writing.

The premise · Simulate mode is where LLD can't reach

LLD teaches structure. System Design needs consequence. The existing engine already runs p50/p95/p99 metrics, cost accounting, chaos events, cascade propagation, narrative hooks. What's missing is pedagogy on top. This batch decides what teaching happens inside the simulation.

🌪
THE WIND-TUNNEL PRINCIPLE
"A wind tunnel doesn't fly the plane. It tells you where the plane is weakest before the plane is in the sky."
Simulate mode is not a game. It is a failure laboratory. The user brings a design. The lab stresses it until it breaks. Every break is a concrete lesson the user could not have learned from a textbook. The simulation is only valuable if the user leaves changed — having updated their mental model of what happens when X fails at scale.
Every metric that moves must have a teaching moment attached. A number without a baseline is noise. A failure without narrative is frustration. A chaos event without context is a lightshow.

Q6 · How do live metrics become teaching signals?

The existing engine already emits p50/p95/p99, throughput, error rate, cache hit ratio, queue depth. Right now they display as numbers. The question: how do we make those numbers pedagogical?

OPTION A · RAW NUMBERS (CURRENT)
Metrics display as-is. User interprets. Example:
p99 latency: 3,200ms
throughput: 1,240 rps
cache hit: 12%
Pros: clean, no opinion. Cons: user has no idea if 3200ms is "fine" or "catastrophic".
⭐ OPTION B · THRESHOLD COACHING
Each metric has known thresholds. System classifies + explains:
⬤ p99: 3,200ms · CRITICAL
SLO violated. User-facing requests timing out. Industry: p99 < 500ms is web norm.

⬤ cache hit: 12% · POOR
88% of requests hit DB. Healthy cache: 85-95%. Your keys may be misshaped.
Pros: every number becomes a lesson. Cons: authoring 40+ metric thresholds.
OPTION C · BASELINE COMPARATOR
Each metric shows alongside "what real systems hit":
Your throughput: 1,240 rps
 └─ Reddit front page: ~15k rps
 └─ Twitter timeline: ~200k rps
 └─ Netflix peak: ~350k rps
 └─ You're at 0.6% of Netflix scale
Pros: gives scale intuition. Cons: baselines go stale, need updates.
OPTION D · B + C COMBINED
Thresholds and baselines together. Each metric has 3 layers: raw number · classification · real-world comparison. Hover reveals deeper chain. Click opens "Why this metric matters" reading.
Pros: maximum teaching density. Cons: 2x authoring work, UI gets busy. Tooltip discipline needed.

Q7 · What simulation activities do we ship?

The engine can run any scenario. We have to pick which framings become first-class "activities" with their own UI, scoring, teaching moments. Pick as many as you want — these are not exclusive.

① VALIDATE
⭐ must-ship
Input: your design from Build mode. Output: "Does this design handle spec?" Runs workload at target scale for 60 seconds, grades against problem's SLO (p99, availability, correctness). Short (60s), low-drama, high-signal. First-time simulation experience.
② STRESS TEST
⭐ must-ship
Find the breaking point. Ramp load from 1x → 100x target. Graph shows where latency hockey-sticks. Teaches "where does my design fall over" viscerally. Result: a number — "your design holds to 14x target before p99 breaches SLO." Generates headline: "You survived 14x. The top 5% solutions survive 50x."
③ CHAOS DRILL
⭐ must-ship · the flagship
Scenario-based narrative. "It's Black Friday 11:58 PM. Traffic 10x in 4 minutes. Then primary-region goes dark. You have 5 minutes to survive." Timed. 1 chaos event every 45-90 seconds. User can intervene (toggle circuit breakers, scale up, switch regions). Post-mortem generated. The flagship simulation.
④ COMPARE · A/B DESIGNS
nice-to-have
Split-screen. Run two designs against the same workload. See which handles chaos better. Example: "Design A: monolith + cache. Design B: microservices + queue. Same traffic, same chaos. Winner?" Teaches tradeoffs viscerally. High production value but limited reuse.
⑤ FORECAST · 3-MONTH HORIZON
later
"Where does your design break next?" Given current design + growth assumption (2x users/month), predicts which component fails first, when, and why. Teaches capacity planning. Advanced — ship after ①②③ are stable.
⑥ ARCHEOLOGY · REPLAY REAL INCIDENTS
ambitious
"What happened at Facebook in Oct 2021?" Rebuild a real outage as a simulation scenario. User lives through the cascade as on-call engineer lived it. Teaches incident response + real systems thinking. Content-heavy — each one is a ~4-hour authoring effort.

Q8 · AI coach during simulation — how intrusive?

Haiku (cheap) or Sonnet (slow) can observe the simulation live and talk. Too much → paternalistic. Too little → user flounders. Pick the vibe.

OPTION A · SILENT WATCHER
No coach during simulation. AI analysis only in post-mortem. User learns by suffering then reading. Classic failure pedagogy (Kapur's productive failure).
Pros: respectful, cheapest. Cons: beginners drown.
⭐ OPTION B · WHISPER MODE
Coach only speaks on teachable moments: first cache-miss storm, first cascade, first timeout fan-out. Max 3 whispers per 5-min simulation. Haiku-powered (cheap, low latency). Ghost-text style, slides in from edge.
"👁 Notice how every request now hits the DB — your cache just evicted. Watch what happens in 10s."
Pros: rare = signal. Cons: moment detection logic to build.
OPTION C · COMMENTATOR
AI narrates continuously like a sports commentator. "Load climbing through 5x · cache hit ratio dropping · DB CPU at 78%..." Maintains tension. Engagement high; signal low. Great for demos.
Pros: cinematic. Cons: Sonnet-grade cost, noisy.
OPTION D · ON-DEMAND CONSULT
Silent by default. User can press ? anytime to pause and ask: "what's happening?" Coach answers with current state. Puts user in control.
Pros: learner-paced. Cons: beginners don't know what to ask.

Q9 · Drill-in features for a running simulation

Pick any that feel necessary. Time-tools turn a flash of red into an autopsy.

① PAUSE & INSPECT ⭐
Space freezes time. Hover any node, see all metrics, in-flight requests, queue depths. Baseline feature.
② TIME SCRUBBER ⭐
Scrub back 30 seconds (live) or entire run (post). "How did we get here?" becomes answerable. Requires ring-buffered state.
③ CASCADE TRACE ⭐
When a failure cascades, visualize the chain: "Cache evicted → DB CPU spiked → write queue backed up → API timeouts → clients retrying → amplification." Each link clickable. The engine already tracks cascades — this exposes them.
④ SLOW-MO
0.25x / 0.5x / 1x / 2x / 4x playback speed. Slow-mo catches sub-second cascades beginners miss at 1x.
⑤ REPLAY & SHARE
Save a simulation run. Replay exactly with annotations overlaid. Export as shareable link — "watch my design survive chaos drill 7". Social moat.
⑥ METRIC DRILLDOWN
Click any metric line on the graph → popover shows "at this moment, DB was at 78%, cache miss spiked 12% → 47%, 3 requests timed out." Traces the local cause.
⑦ "WHAT IF?" BRANCHING
Pause mid-simulation, edit the design, resume — see if the change would have saved you. Counterfactual learning. Powerful but expensive (requires state-cloning).

Q10 · How does Simulate link back to Learn and Drill?

A simulation that ends in "you died" with no path forward is a dead-end. The loop must close.

OPTION A · POST-MORTEM ONLY
After run, generate a report. Links to relevant concept pages and drill problems. User clicks to dig deeper on their own initiative.
⭐ OPTION B · TRIPLE LOOP
Three explicit paths out of a simulation result:
📖 Learn path:
"You failed on cache eviction. Read: Caching Strategies."
🎨 Build path:
"Want to try a different design? Back to Build, load your attempt, iterate."
⏱ Drill path:
"Try 'Design Twitter' — same cascade pattern, different constraints."
Each path opens with context preloaded.
OPTION C · FORCED REMEDIATION
Can't retry the simulation until you've read the remedial concept. Feels nanny-ish but guarantees learning. Good for Rookie persona, kill-switch for Architect persona.
OPTION D · CHALLENGE ESCALATION
Pass simulation → unlock harder chaos drill. Fail → unlock easier variant. Difficulty auto-tunes. Gamified but maintains dignity (no "you failed" shaming).

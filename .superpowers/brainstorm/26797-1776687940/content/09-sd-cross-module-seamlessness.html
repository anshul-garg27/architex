The premise · Thirteen modules or one organism?

Architex has 13 modules. LLD and SD are the flagship two. Algorithms, Databases, Networking, Data Structures, Graphs, Theory, Observability, etc. fill the rest. Today a user clicking "SD" feels like they entered a different product. The seamless ideal: curiosity flows without friction. Reading about Consistent Hashing in SD, one click takes you to the LLD pattern that implements it, and another click opens a database module deep-dive on consistent sharding. One brain.

🧠
THE ONE-ORGANISM PRINCIPLE
"The user should never have to remember which module a concept lives in — only what they want to learn next."
Concepts, patterns, problems, and failures reference each other across modules. Progress earned in one module carries over. Profile tracks everything. When the user switches modules, state is preserved and context is maintained. This batch decides how deep the seams disappear.

Q35 · Knowledge graph depth — how linked is everything?

From Batch 2 we agreed cross-linking is deep. Now: how does the user see and navigate that graph?

OPTION A · TEXT LINKS ONLY
Every concept page has inline links to related ones. "See also: LLD Observer Pattern." Simple, cheap, text-based.
⭐ OPTION B · LINK CARDS + NEIGHBOR SIDEBAR
End-of-page bridge cards (from Batch 8) + a permanent "Related" sidebar on every page showing top-6 related items across modules, weighted by relationship strength. Always visible; 2-click traversal to anywhere.
Related to Consistent Hashing:

🏛 SD · Design Dynamo / Cassandra
💾 DB · Sharding strategies
🌐 NET · Load balancer routing

OPTION C · VISUAL GRAPH EXPLORER
Dedicated Knowledge Graph page: force-directed graph of all 700+ content pieces. User clicks node → content opens. Zoom to explore clusters. Beautiful, but search-first users never find it.
OPTION D · B + C
Sidebar on every page (B) + optional graph explorer view (C) accessible via ⌘G. Power users love the graph; everyone benefits from the sidebar.

Q36 · Unified progress model

Earn progress in SD — does it count toward LLD? FSRS cards from one module appear in another's review queue?

OPTION A · ISOLATED
Each module tracks its own progress. User has LLD streak vs SD streak vs Algorithms streak. Clean but fractures identity.
⭐ OPTION B · ONE PROFILE, PER-MODULE LENSES
Single user profile holds all progress. Dashboard shows per-module breakdowns. Streaks are cross-module ("you learned something somewhere today"). FSRS queue mixes all modules. Badges are global.

Q37 · Universal command palette & shortcuts

One keystroke to go anywhere — the power-user's love letter. Multi-select features to ship.

① ⌘K COMMAND PALETTE ⭐
Universal. Search across: concepts, problems, patterns, chaos events, incidents, templates, past attempts, settings. Type observer → find LLD Observer pattern + SD pub-sub concept + related drills.
② ⌘, PROFILE ⭐
Opens your unified profile: streaks, cards-due, path progress, last-sessions-per-module.
③ ⌘⇧M MODULE SWITCHER ⭐
Jumps between modules preserving current mode. If in LLD Drill, pressing this + picking SD puts you in SD Drill. Muscle memory transfers.
④ ⌘G KNOWLEDGE GRAPH
Opens the visual graph (if Q35=D). From anywhere. Always lands on current-page's node.
⑤ ⌘R REVIEW QUEUE ⭐
One-keystroke review: jumps to cross-module FSRS queue. Can be done for 5 minutes before starting any work.
⑥ ⌘? SHORTCUTS SHEET
Overlay with all shortcuts grouped by mode/module. Context-aware (shows Drill keys when in Drill).
⑦ NATURAL LANGUAGE QUERIES
⌘K + question mark enables AI-backed search. "Show me problems that use consistent hashing and have multi-region chaos events." Returns filtered list.
⑧ RECENTS STACK
⌘[ / ⌘] navigate back/forward across content. Like browser history but cross-module aware.

Q38 · State persistence across mode + module switches

Mid-drill, user switches to read a concept. Comes back. Does state survive?

OPTION A · LOSE STATE ON SWITCH
Leaving a drill = abandoning it. Canvas cleared. Timer reset. Harsh but simple.
⭐ OPTION B · DB-FIRST PERSISTENT
All in-progress state (drill canvas, timer state, simulation run, learn scroll position) saves to DB every ~10s. Cross-device resume. Cross-browser resume. Matches LLD's decision (Q8 in LLD brainstorm).
Returning to drill:
"Welcome back. You were 18:24 into Design Netflix (timed mock, The Staff Engineer). Resume where you left off?" [Yes] [Abandon]
OPTION C · LOCAL-ONLY (localStorage)
Persist per-browser only. No cross-device. Faster, no server load. Matches "offline first" aesthetic but hurts mobile-to-desktop transitions.
OPTION D · B + "SNAPSHOT" EXPLICIT
DB persistence (B) PLUS explicit "Save snapshot" for nameable save-points. Useful for ambitious designs user wants to fork from.

Q39 · Daily re-entry experience

User opens Architex in the morning. What do they see?

OPTION A · LAST MODE
Lands where they left off. Zero friction, zero nudge. Power-user default.
⭐ OPTION B · DAILY DASH
Homepage with: cards due today · streak · "pick up where you left off" · "one thing I'd recommend" (AI) · recent community. Honors unified profile.

OPTION C · FOCUS MODE ENTRY
User picks a goal ("30-min review" / "full drill" / "just read"). Session shaped accordingly. Anti-rabbit-hole.
OPTION D · B DEFAULT, A TOGGLE
Dashboard by default, "Always resume last" toggle in profile for power users. Best-of-both.

Q40 · Shared AI context across modules

AI-assisted features exist in LLD (explain, tutor), SD (coach, interviewer), possibly elsewhere. Does the AI know what you did in other modules?

OPTION A · NO SHARED CONTEXT
Each AI invocation is stateless. Cheap. But can't notice "you just read this concept — let me reference it."
⭐ OPTION B · PROGRESS-AWARE
AI sees user's module-wide progress: recent reads, mastery levels, common weak points. Sonnet prompts include a compact "user context" block. Not full conversation history — just structured state.
user-context: {
  recent: ["sd/consistent-hashing", "lld/observer"],
  mastery: { caching: 0.7, sharding: 0.4 },
  weak: ["distributed-consensus"],
  streak: 7, cards_due: 8
}
OPTION C · FULL CONVERSATION MEMORY
AI remembers actual conversation turns across modules. "As we discussed earlier in LLD..." Costly (token bill), privacy-complex, powerful.
OPTION D · USER-CONTROLLED MEMORY
User picks per session: "remember me" vs "fresh". Privacy honored. Adds UX friction. For trust-conscious users.

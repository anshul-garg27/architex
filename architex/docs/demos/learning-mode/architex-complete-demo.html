Step 1 Complete

THE PROBLEM

Imagine subscribing to a YouTube channel. You don't refresh the page every minute -- you get notified when there's a new video. That's the Observer pattern.

Real-world analogy: A newspaper delivers to all subscribers when a new edition is published. Subscribers don't poll the printing press.
Step 2 Complete

MEET THE SUBJECT

The Subject holds state and a list of observers. When something changes, it iterates through its observers and calls their update() method.

It has three key methods: attach(), detach(), and notify().

Step 4 - Current

THE CONTRACT

The Observer interface is the contract. Any class that implements it can subscribe to a Subject. This is the key to loose coupling -- the Subject doesn't know (or care) what concrete type its observers are.

interface Observer {
update(data: any): void;
}

On the canvas, the Observer interface is now highlighted. Notice how Subject references Observer -- not any concrete class.

Primary Feedback
Missing detach() method on the Subject class. Without it, observers can never unsubscribe, leading to potential memory leaks and unwanted notifications. This also violates the principle of symmetric operations -- if you can attach(), you should be able to detach().
Suggestion
Consider adding error handling in notify() -- if one observer throws, the rest still need to be notified. A try/catch inside the iteration loop would make this more robust.

What is the key difference between Observer and Pub/Sub?
Click to reveal answer
Answer
Observer: The Subject directly notifies its observers. They have a direct reference -- the Subject knows its observers exist.

Pub/Sub: A message broker (event bus) mediates. The publisher and subscriber have zero knowledge of each other. This adds a level of decoupling but also complexity.

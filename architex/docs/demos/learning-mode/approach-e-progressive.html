The Problem
Imagine you subscribe to a YouTube channel. You don't check every hour for new videos — you get notified when one drops. Without this, every subscriber would have to keep refreshing the page. That's polling. It's wasteful.

The Observer pattern solves exactly this: let interested parties register themselves, and notify them all when something changes. No polling. No tight coupling. Just clean, event-driven communication.
2
Meet the Subject
The Subject is the thing being watched. It maintains a list of everyone who cares about its state. Three core responsibilities:

attach() — add a new observer
detach() — remove one
notify() — tell all observers something changed

The Contract
The Observer interface defines the contract: any class that implements update() can subscribe to the Subject.

This decoupling is the pattern's superpower — the Subject doesn't need to know what its observers are, only that they implement update().

Concrete Observers
Now we add the real implementations. ConcreteObserverA might display the data, while ConcreteObserverB logs it. Both implement update() from the Observer interface.

This is the power of polymorphism — the Subject calls notify() once, and each observer reacts in its own way. Add a new observer type? Zero changes to Subject.
7
The Code
Here's the full TypeScript implementation:
observer-pattern.ts 
interface Observer {
  update(subject: Subject): void;
}

class Subject {
  private observers: Observer[] = [];
  private state: any;

  attach(o: Observer) {
    this.observers.push(o);
  }

  detach(o: Observer) {
    this.observers = this.observers
      .filter(obs => obs !== o);
  }

  notify() {
    // Each observer reacts in its own way
    for (const obs of this.observers) {
      obs.update(this);
    }
  }

  setState(s: any) {
    this.state = s;
    this.notify(); // auto-notify
  }
}
8
Key Takeaway
  • Loose coupling — Subject and Observers know almost nothing about each other
  • Open/Closed Principle — add new observers without modifying Subject
  • Event-driven — no polling, no wasted cycles, react when it matters
  • Real world — DOM events, RxJS, React state, pub/sub systems all use this pattern

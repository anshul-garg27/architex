<!-- Section: itch -->
## The Itch

You start your service and it boots three database pools, two Redis clients, and a metrics collector. Each one allocates its own socket pool and each thread you spin up has to know which pool to grab. By the time your traffic doubles, your memory graph is full of copies of things that should be singular. You need a way to say — at the class level, not just the documentation level — *exactly one of this*.

Singleton is the GoF pattern for that guarantee.

<!-- Section: definition -->
## Definition

A Singleton makes a class responsible for *both* its single instance and the means of access to it. Creation is hidden; lookup is explicit.

Two parts must both be true:

1. No caller may construct the class from outside.
2. A well-known static accessor returns the same reference every time.

Everything else — lazy vs. eager, locking strategy, serialization — is variation on those two rules.

<!-- Section: mechanism -->
## Mechanism

The dance is short. Each step exists for one reason.

1. The constructor is hidden so nobody can slip a `new` in. Most languages mark it `private`.
2. The class keeps a static slot — the one place the instance will ever live.
3. A public static accessor — `getInstance` by convention — returns the slot.
4. If creation is lazy, the accessor has to synchronize or otherwise guarantee that two callers can't both win the race.

In Java, that last step is almost always either eager static initialization or a nested *holder* class that lazy-loads on first call without locking.

<!-- Section: anatomy -->
## Anatomy

Only one class is strictly required — <Class id="Singleton" /> itself. In practice it carries:

- A private or protected constructor
- A private static field holding the instance
- A public static `getInstance` method

Some variants extract the state into a separate class to enable mocking, but the pattern is still a single stable identity.

<!-- Section: numbers -->
## Numbers

A reference lookup is O(1). The instance's footprint is whatever state it carries; the Singleton wrapper itself costs a reference.

On the JVM, eager-initialized class-holder idiom is free — the class is loaded exactly once and the holder runs under the class loader's built-in lock. Double-checked locking costs a `volatile` read per access, which is a single memory fence.

<!-- Section: uses -->
## Uses

The JDK's `Runtime.getRuntime()` is the canonical Singleton. Kafka's consumer-group coordinator must be process-wide for exactly-once offset commits. Gradle's project graph is the single source of truth during a build.

All three share a property: *multiple copies would be wrong*, not just wasteful.

<!-- Section: failure_modes -->
## Failure Modes

The most common failure isn't technical — it's testability. Tests mutate the instance and bleed state between cases. Give your Singleton a `reset()` for tests or inject it via constructor.

The second common failure is unsafe lazy initialization. Without `volatile` (or equivalent), a partially-constructed object can leak to a concurrent reader.

The third is serialization: deserializing a Singleton quietly creates a second instance unless you override `readResolve()`.

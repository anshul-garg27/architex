<!-- Section: itch -->
## The Itch

You ship a cross-platform app. On macOS you need a MacButton + MacWindow + MacMenu; on Windows a WinButton + WinWindow + WinMenu. They must match — a MacButton inside a WinWindow is visually broken. You need a way to hand callers a *set* of matching components.

<!-- Section: definition -->
## Definition

Abstract Factory packages related creation operations into a single interface. Callers ask the <Class id="AbstractFactory" /> for products; each <Class id="ConcreteFactory" /> returns products from one coherent family.

<!-- Section: mechanism -->
## Mechanism

1. Resolve which <Class id="ConcreteFactory" /> is active (typically at startup).
2. Store it somewhere callers can reach.
3. Callers use factory methods on the interface; concrete types are invisible.

<!-- Section: anatomy -->
## Anatomy

Four roles: <Class id="AbstractFactory" />, <Class id="ConcreteFactory" />, <Class id="AbstractProduct" />, <Class id="ConcreteProduct" />. Each family has its own ConcreteFactory + a parallel set of ConcreteProducts.

<!-- Section: numbers -->
## Numbers

Minimum four classes; typically more as product count grows. Memory is one factory instance per scope.

<!-- Section: uses -->
## Uses

Java AWT Toolkit (Swing, SWT). Cross-platform test doubles (MemoryStorage vs SqlStorage). Dependency injection frameworks use Abstract Factory under the hood for scoped object graphs.

<!-- Section: failure_modes -->
## Failure Modes

Product-family rigidity is the biggest risk — adding a new product to the family requires updating every ConcreteFactory. Counter with default methods where possible.

The second is kit mismatch: callers accidentally pull products from different factories and wire them together. Generics or factory-per-scope prevent this.

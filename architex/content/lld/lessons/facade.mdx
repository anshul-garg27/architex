<!-- Section: itch -->
## The Itch

Your payments service ships a checkout endpoint. It imports six internal SDKs: `pricing`, `inventory`, `tax`, `fraud`, `payments`, `notifications`. Each SDK has its own init pattern, its own config file, its own error taxonomy. Every new feature opens a file, writes fifty lines of boilerplate â€” fetch pricing, reserve inventory, compute tax, run fraud, authorize, fallback on decline, notify â€” then closes the file. You now have forty-seven callers, each repeating the same sequence with small variations, each with its own subtle bug.

Then `payments-sdk` ships v4 with a breaking signature on `authorize()`. You grep for callers. There are forty-seven. Forty-seven pull requests.

Facade is the pattern for the door you wish had been there all along. One class. One method. `checkout(cart, customer)` returns a result. The six subsystems are still there â€” still testable, still reachable â€” but no caller mentions them. The next SDK break is one pull request, not forty-seven.

You've seen this before. jQuery was this over the DOM. `fetch()` was this over `XMLHttpRequest`. Stripe Elements is this over forty calls of PCI compliance. BFFs are this over a mesh of microservices. The pattern has a name.

<!-- Section: definition -->
## Definition

> *Provide a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use.*
>
> â€” Gang of Four, *Design Patterns* (1994), p.185

Two obligations make a Facade:

1. It sits *over* a subsystem of collaborating classes â€” two or more, usually several.
2. It offers the caller a *simpler* surface than the subsystem itself exposes.

The subsystem keeps working; its classes remain usable. The Facade is convenience, not prison. If you ever find yourself routing through the Facade because the subsystem classes are hidden from you, you're in a Facade-turned-gatekeeper situation that the GoF were explicit about not requiring.

This is the pattern that is most often confused with its neighbors. Adapter translates *one* mismatched interface. Proxy preserves the same interface and adds behavior. Mediator wires peers together. Facade simplifies access to many. Keep those four sentences in your head and most interview traps resolve themselves.

<!-- Section: mechanism -->
## Mechanism

The work is almost physical. You draw a ring around the subsystem classes your callers are currently reaching into. Everything inside the ring is going to be hidden. Everything outside has to ask the Facade for what it wants.

Then you forget the subsystem's API for a moment and write down what the caller is actually *trying to do*. Not `buildCryptogram()` then `authorize()` then `settle()` â€” but `chargeCard(amount, card)`. The Facade's method names come from the caller's vocabulary, not the subsystem's.

<Class id="Facade" /> holds references to the <Class id="Subsystem" /> classes. Each Facade method forwards to a short sequence of subsystem calls â€” orchestration only, no real work. If the Facade starts implementing domain logic, it has stopped being a Facade and become something else (usually a God-object; see Failure Modes).

The subsystem classes stay *usable*. A power user who needs raw access bypasses the Facade and talks to the subsystem directly. This is a deliberate property of the pattern, not a leak. If you take away the bypass you've built a Gateway, not a Facade.

<!-- Section: anatomy -->
## Anatomy

Three roles, most lessons collapse them to two.

<Class id="Facade" /> is the one new class the pattern introduces. It exposes the narrowed, caller-friendly interface. Its implementation is almost entirely delegation.

<Class id="Subsystem" /> stands in for two or more existing classes that were already doing the work. They don't change. They don't know about the Facade â€” one-way coupling, always. The Facade depends on the subsystem; the subsystem has no reason to reference the Facade.

<Class id="Client" /> is the caller. After the Facade lands, the Client imports only the Facade. The subsystem becomes an implementation detail.

One detail catches people: the Facade is *not* an abstract class or interface by definition. It's usually a concrete class. An interface layer is a refinement (useful for testing), not part of the base structure.

<!-- Section: numbers -->
## Numbers

Reach for Facade when:

- **At least two distinct subsystem classes** would otherwise be called together. One subsystem class behind a Facade is usually wasted indirection â€” just fix the class.
- **Three to eight subsystem method calls** collapse into one to three Facade methods. The ratio is the point of the pattern.
- **Blast radius of subsystem change** drops from *O(callers)* to *O(1)* when the Facade is the only caller of the subsystem.

Stripe Elements is the textbook case: roughly forty PCI-compliance calls â€” element mounting, tokenization, 3DS challenge, Strong Customer Authentication, retry, fallback â€” collapse into one `<PaymentElement/>` component plus one `confirmPayment()` call. The subsystem classes still exist; the Stripe dashboard and raw-API users still reach them. Most callers never do.

There is no runtime cost to Facade beyond one extra function call per operation â€” O(1), inlined by any real compiler. The real cost is organizational: someone has to own the Facade and evolve it alongside the subsystem.

<!-- Section: uses -->
## Uses

Every developer alive uses Facades daily, usually without naming them.

**jQuery** is the canonical user-land example. `$('#foo').fadeIn()` hides `document.getElementById` (or `querySelector`), a cross-browser event model, `requestAnimationFrame` loops, computed-style reads, and transition events. For a decade it was the only tolerable way to talk to the DOM. When `querySelector` and `fetch` finally shipped, much of jQuery's value disappeared â€” the DOM had absorbed the Facade.

**`fetch(url)`** is itself a Facade over the `XMLHttpRequest` lifecycle: `open`, `setRequestHeader`, `onreadystatechange`, `responseText` parsing, abort handling â€” collapsed into one promise-returning call with a response object.

**Stripe Elements** hides ~40 PCI-compliance, 3DS, and retry calls behind a single widget. You cannot pass card numbers through Stripe's Facade and fail PCI; the whole point is that the simpler API is also the safer one.

**Backend-for-Frontend (BFF)** is the architectural scale version. A single mobile-shaped endpoint fans out to 6-12 microservices so the client never round-trips each one. The mobile team owns the BFF; it's their Facade over the service mesh.

**AWS SDK** high-level clients (`s3.putObject`, `ddb.getItem`) are Facades over SigV4 signing, endpoint resolution, multipart chunking, retry backoff, and HTTP. Raw signing is still possible; nobody does it.

**React 18's `createRoot(container).render(<App/>)`** replaces a decade of `appendChild` + manual reconciliation wiring + hydration edge cases with one factory plus one method. The reconciler is still there â€” React's internals still reach it â€” but the Client never does.

One property is shared across all six: the Facade adds no capability. Every call it makes was possible before. What it adds is the right *shape* for the caller.

<!-- Section: failure_modes -->
## Failure Modes

The first and worst failure is the **God-object Facade**. A team adds a Facade, then adds "just one more method" to it for months. Eighteen months in, the Facade has 240 methods, tests that take 20 minutes, and every caller reaches through it for everything. It has become the application, not a door to the subsystem. The fix is almost always to split by *caller intent*: a Facade for checkout, a different Facade for admin, a third for reporting, all over the same subsystem. Two Facades over one subsystem is fine. One Facade over eighty methods is a smell.

The second, quieter failure is the **leaky abstraction**. Your Facade method returns a raw subsystem type â€” a `ByteBuffer`, an internal `PaymentContext` with twelve nullable fields â€” because returning the "real" thing was the easiest first move. Now every caller depends on that type. The moment you try to evolve the subsystem, you break everyone. The rule: return plain data shapes the caller owns. If a subsystem type crosses the Facade boundary, you've committed to it as public API â€” be explicit about that choice.

The third is the **1% case**. A power user needs one subsystem knob the Facade doesn't expose. They patch the Facade (bloating it) or bypass it (defeating it). Neither is wrong â€” but the right answer is documented. Keep the subsystem classes accessible and tell users, clearly, when to reach past the Facade. The Facade is a convenience, not a prison.

The fourth is **stale-by-design**. The subsystem team ships a feature. The Facade team is separate, and lags by a quarter. For three months, the Facade is the *worst* way to use the subsystem â€” missing flags, stale defaults â€” and callers start going around it. The fix is ownership: same team owns the Facade and the subsystem, or Facade parity is in the subsystem's definition of done. Otherwise the Facade rots.

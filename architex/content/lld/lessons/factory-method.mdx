<!-- Section: itch -->
## The Itch

Your checkout page must support Stripe today. Next quarter you're adding Adyen. Next year, Razorpay. Hardcoding `new StripeProcessor()` in your checkout flow means every new provider rewrites the flow. You need a way to pick the concrete processor *without* the callers caring which they got.

<!-- Section: definition -->
## Definition

Factory Method sits between "ask the class to construct" and "subclass picks the actual class". A base class defines the *shape* of the create call; subclasses plug in the concrete <Class id="Product" /> they produce.

<!-- Section: mechanism -->
## Mechanism

The <Class id="Creator" /> declares `factoryMethod()` returning the abstract <Class id="Product" />. Each <Class id="ConcreteCreator" /> overrides the method to return a specific <Class id="ConcreteProduct" />.

Callers interact only with Creator and Product types. They never see the concretes. Switching implementations is a matter of passing a different Creator instance.

<!-- Section: anatomy -->
## Anatomy

Four classes form the minimal structure. In practice, you often have many ConcreteCreator/ConcreteProduct pairs.

<!-- Section: numbers -->
## Numbers

Minimum four classes. Extending to a new product type adds two classes (one Creator, one Product) — linear growth, not explosive.

<!-- Section: uses -->
## Uses

Stripe's PaymentMethod model: each method has its own creator but the checkout code sees only `PaymentMethod`. Spring's BeanFactory is a generalized Factory Method where creators are configured externally.

<!-- Section: failure_modes -->
## Failure Modes

Class explosion is the first headline risk — each new product drags along a new Creator. The second is factory-of-factories — don't build three layers of creators when one Simple Factory or parameterized factory would do.

<!-- Section: itch -->
## The Itch

You add `new Request(url, headers, body, timeout, retries, backoff, tlsConfig, proxyUrl…)` and the parameter list is now longer than the function body that uses it. Each new feature adds one more arg. You can't remember the order. Builder gives you named-parameter semantics without needing the language feature.

<!-- Section: definition -->
## Definition

Builder separates the *construction* of a complex object from its *representation*. Two callers can use the same Builder and receive different final objects depending on which steps they trigger.

<!-- Section: mechanism -->
## Mechanism

1. The caller gets a <Class id="Builder" /> instance.
2. Calls step methods in any order (most builders allow partial ordering).
3. Calls `build()` to receive the final <Class id="Product" />.

A <Class id="Director" /> is optional — used when multiple callers need the same construction sequence.

<!-- Section: anatomy -->
## Anatomy

Three classes in the base form, four with Director. Builder returns `this` from each step for fluent chaining.

<!-- Section: numbers -->
## Numbers

Builders typically have 5-25 step methods. Build time is O(n) in the number of steps.

<!-- Section: uses -->
## Uses

java.lang.StringBuilder. OkHttp's Request.Builder. Apache HttpClient's RequestBuilder. Gradle's `DependencyHandler`. All of them exist because the underlying product has many optional configurations.

<!-- Section: failure_modes -->
## Failure Modes

The most common mistake is forgetting `build()` — callers configure and never call the terminal method. Second: reusing a mutable builder across threads or across products. Third: confusing fluent chaining style with the Builder *pattern* (not every fluent API is a Builder).

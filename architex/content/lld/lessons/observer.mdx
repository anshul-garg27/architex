<!-- Section: itch -->
## The Itch

You have a stock-ticker app. Five widgets on screen — chart, sidebar, alert, trade form, history — all need the current price. Option A: hardwire each widget to the data source. Option B: let the data source broadcast, widgets subscribe. Option B scales; Option A rewrites the data source every time a widget is added.

<!-- Section: definition -->
## Definition

Observer declares a one-to-many relationship: one <Class id="Subject" />, many <Class id="Observer" />s. The Subject notifies observers when its state changes; observers decide how to react.

<!-- Section: mechanism -->
## Mechanism

1. Observers call `subject.attach(this)`.
2. When state changes, Subject calls `observer.update(this)` on each attached observer.
3. Observers may call `subject.getState()` to pull details (pull model), or Subject may pass the delta as a parameter (push model).
4. Observers call `subject.detach(this)` to stop receiving updates.

<!-- Section: anatomy -->
## Anatomy

Four roles form the classic structure. In practice the Subject and Observer are often interfaces with multiple concrete implementations.

<!-- Section: numbers -->
## Numbers

Notify cost is O(k) in the number of observers. Subscribe/unsubscribe is O(1) with a hash set, O(n) with a list.

<!-- Section: uses -->
## Uses

Java's `PropertyChangeSupport`. Node.js's `EventEmitter`. Rx/React signals. Git hooks are a Subject notifying filesystem-level observers.

<!-- Section: failure_modes -->
## Failure Modes

Leaked subscriptions are the #1 Observer bug — long-lived subjects retain observers past their lifetime. Return an unsubscribe handle from attach().

Notification order is not guaranteed by the GoF spec. If order matters, document it and enforce it in tests.

Reentrancy — an observer that triggers another state change mid-notify — will blow up if you iterate the observer list in place. Copy before iterating.

<!-- Section: itch -->
## The Itch

You have a game with 500 enemy variants. Each differs from a "base zombie" by 2-3 stats. Hand-coding 500 constructors means 80% repeated configuration. What you want: start from a base instance, tweak the diff, get a new instance.

<!-- Section: definition -->
## Definition

Prototype delegates object creation to the object itself via a <Class id="Prototype" />.clone() method. Callers don't name the class — they name the *instance to copy*.

<!-- Section: mechanism -->
## Mechanism

1. Declare <Class id="Prototype" /> with `clone()`.
2. <Class id="ConcretePrototype" /> implements clone with appropriate copy depth.
3. <Class id="Client" /> calls `somePrototype.clone()` and receives a fresh instance.
4. Often a registry holds named prototypes for lookup by key.

<!-- Section: anatomy -->
## Anatomy

Two classes in the minimal form. Three if you add a Client that holds the prototype registry.

<!-- Section: numbers -->
## Numbers

Shallow clone is O(1) in pointer copies. Deep clone is O(n) in object graph size. Measure: never assume.

<!-- Section: uses -->
## Uses

JavaScript's `Object.create`. CAD applications cloning component templates. Game engines cloning entity archetypes. Kubernetes cloning pod specs with overrides.

<!-- Section: failure_modes -->
## Failure Modes

Shallow-vs-deep is the #1 bug source — a clone that shares a nested array tangles original + copy. Track every mutable field and deep-clone it, or use an immutable data structure.

Circular refs blow the stack unless you memoize. Add a visited-set to the clone recursion.

Clone drift happens when someone adds a new field and forgets to update clone(). Use reflection or a library that handles this automatically.
